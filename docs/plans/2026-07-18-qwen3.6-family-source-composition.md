# Qwen3.6 Family Source Composition Plan

> Status: active implementation plan for extracting the Qwen3.6 family-owned C++ leaves from the
> registered 27B target and composing family and exact-target sources directly into `ninfer_engine`.
>
> Scope: repository/source ownership, CMake source composition, the family Frontend and prepared
> values, frontend-resource binding, passive common Vision definitions, the 27B package facade,
> Engine prepared-prompt storage, affected tests/tools, and active architecture documentation.
>
> Prerequisite: the current registered 27B frontend behavior and its six embedded frontend resources
> are the accepted Qwen3.6 family contract. This plan does not change those resource contents or add
> comparison, provenance, version, or compatibility machinery around them.

## 1. Required outcome

After this plan is complete:

- `src/targets/qwen3_6/` is the physical owner of the Qwen3.6 family Frontend, owning
  `PreparedPrompt`, `PreparedPromptData`, `OutputSession`, frontend-resource vocabulary, and passive
  common Vision geometry/binding definitions;
- `src/targets/qwen3_6_27b_rtx5090/` no longer contains a frontend implementation and continues to
  own its exact artifact package, Text/MTP bindings, target-width Vision projection, Program,
  schedules, state, workspace, Op/fusion choices, CUDA Graph policy, and diagnostics;
- the target layer creates neither a family static library nor an exact-target static library;
  family and exact-target `.cpp` files are explicit source contributions compiled exactly once into
  the existing `ninfer_engine` product target;
- the 27B package aliases the family `Frontend`, `PreparedPrompt`, and `OutputSession` types while
  retaining exact `LoadPlan`, `LoadedModel`, `SequencePlan`, `RequestPlan`, and `Program` types;
- the public opaque `ninfer::PreparedPrompt` directly owns one `qwen3_6::PreparedPrompt`; the
  exact-target prepared-value variant, target tag, provenance, and mismatch branch do not exist;
- the registered 27B product retains its current Text, image/video Vision, MTP, prefix-reuse, CLI,
  serving, diagnostic, and benchmark behavior;
- no 35B runtime target, source directory, CMake source contribution, registry alternative, Program,
  schedule, state, graph, or Engine route is added.

This is a source-ownership and composition cutover. The old target-private frontend placement and
the old exact-target library target are removed rather than retained as compatibility paths.

## 2. Fixed decisions

### 2.1 One compile owner for registered target code

`ninfer_engine` is the sole CMake target that compiles family leaves and registered exact-target
sources. The target subdirectories contribute explicit source lists with `target_sources`; they do
not call `add_library`.

Do not introduce any of the following:

- `ninfer_qwen3_6_family` as a static, object, interface, or shared library;
- `ninfer_qwen3_6_27b_rtx5090` as a static, object, interface, or shared library;
- a generic target-registration CMake macro;
- source globbing or directory discovery;
- compiling the same family translation units separately into each exact target;
- linking one exact target through another exact target.

The existing mechanism libraries such as `ninfer_core`, `ninfer_artifact`, `ninfer_ops`,
`ninfer_text`, and `ninfer_media_decode` are not reorganized by this plan.

### 2.2 Family is a source owner, not a target package

`src/targets/qwen3_6/` has no `package.h`, target key, `model_id`, registry entry, `Program`, layer
loop, schedule, mutable sequence state, workspace owner, CUDA Graph, Op dispatch, or fallback path.
Its `export/` headers are repository-internal cross-directory contracts, not installed product API.

### 2.3 Exact targets remain execution owners

The family directory may own represented values and immutable binding vocabulary. It may not decide
when or how Text, Vision, or MTP executes. In particular, the following remain independently
implemented in every exact target even when the high-level architecture is related:

- layer count and the realized three-GDN/one-full-attention layer loop;
- Text and MTP weight structures;
- dense versus MoE scheduling and weight views;
- Vision call ordering and composition with Text;
- Op selection, fixed-shape route selection, and fusion boundaries;
- request planning, workspace reuse, prefix policy, recurrent/MTP state, and CUDA Graph policy.

The small full-attention-layer predicate may remain duplicated in exact-target code. It is part of
the target's realized topology and does not justify a family graph abstraction.

### 2.4 Direct family prompt ownership

Both the current package and a later sibling package use exactly the same family prompt types. Do
not add an exact-target discriminator, resource fingerprint, Engine-instance provenance, family
variant, or cross-target mismatch check. The supported route is one Engine preparing and consuming
its own request.

### 2.5 No resource comparison lane

Each artifact supplies the six resources consumed by the same family frontend. The family resource
plan binds those six canonical objects and produces one `FrontendResources` value. It does not
compare one target's resources with another target's resources and does not accept a target identity
parameter. Ordinary parsing of the represented frontend contract remains part of frontend behavior;
this plan adds no second identity or compatibility validation layer.

## 3. Explicit non-goals

Do not use this work to:

- register or partially scaffold `qwen3_6_35b_a3b_rtx5090`;
- change any checkpoint file, `.ninfer` object name, tensor inventory, format, layout, quantization,
  weight bytes, frontend-resource bytes, or conversion recipe;
- change token rendering, tokenization, detokenization, image/video preprocessing, MRoPE positions,
  default stop behavior, or output-channel behavior;
- redesign the public `Engine`, prompt-input, serving, OpenAI, or Anthropic APIs;
- share Text, GDN, full-attention, MTP, dense MLP, sparse MoE, sampling, or Vision execution graphs;
- move a target-specific CUDA implementation out of `src/ops` or create a family Op layer;
- add generic model configs, family base classes, virtual dispatch, graph IR, plugin discovery,
  runtime target discovery, or a source/binary compatibility path;
- perform unrelated target cleanup, performance tuning, kernel work, or test expansion.

## 4. Current ownership to migrate

The current 27B target mixes four distinct responsibilities that this plan separates:

| Current location | Current content | Destination/steady state |
|---|---|---|
| `export/.../package.h` | exact load/program facades plus `Frontend`, `PreparedPrompt`, and `OutputSession` | exact facades remain; the three frontend facades move to `qwen3_6/export/.../frontend.h` and become aliases |
| `impl/frontend/` | tokenizer, template renderer, media processor, MRoPE prompt construction, output decoder, prepared data | move as one behavior-preserving family implementation |
| `impl/load/bindings.*` | six frontend resources, all Text/MTP bindings, complete Vision bindings, materialized views | split family frontend/common-Vision plans from exact Text/MTP/merger-output bindings |
| `impl/config.h` | exact Text facts plus common Vision facts | keep Text/topology exact; move Vision-backbone facts to family `vision.h` |
| `targets/registry.h` | exact target variant and an exact prepared-value variant | retain only the exact target variant; delete the prepared-value variant |
| `runtime/engine/engine.cpp` | public prepared envelope wrapping the exact prepared-value variant | directly own one family prepared value |
| target `CMakeLists.txt` | independently buildable exact-target static archive | explicit `target_sources(ninfer_engine ...)` contribution |

No source file is moved merely because two targets might someday use it. A file moves only when its
complete invariant matches the family boundary in Section 2.

## 6. CMake source-composition design

### 6.1 Root construction order

In `src/CMakeLists.txt`, create `ninfer_engine` after its mechanism dependencies are declared but
before adding target subdirectories:

```cmake
add_library(ninfer_engine STATIC
  runtime/engine/engine.cpp
  targets/registry.cpp)

ninfer_internal_includes(ninfer_engine)
target_include_directories(ninfer_engine PUBLIC ${PROJECT_SOURCE_DIR}/include)

add_subdirectory(targets/qwen3_6)
add_subdirectory(targets/qwen3_6_27b_rtx5090)

target_link_libraries(ninfer_engine PRIVATE
  ninfer_artifact
  ninfer_core
  ninfer_ops
  ninfer_text
  ninfer_media_decode)

add_library(ninfer::engine ALIAS ninfer_engine)
```

The exact ordering of the calls may follow existing CMake style, but these properties are required:

- `ninfer_engine` exists before a target fragment calls `target_sources`;
- family sources are added once;
- every registered exact target adds only its own sources;
- all dependency archives appear on the final engine link interface once;
- `ninfer::engine` remains the product alias.

### 6.2 Family CMake fragment

`src/targets/qwen3_6/CMakeLists.txt` contains only explicit source/include contributions:

```cmake
target_sources(ninfer_engine PRIVATE
  ${CMAKE_CURRENT_SOURCE_DIR}/impl/frontend/chat_template.cpp
  ${CMAKE_CURRENT_SOURCE_DIR}/impl/frontend/frontend.cpp
  ${CMAKE_CURRENT_SOURCE_DIR}/impl/frontend/processor.cpp
  ${CMAKE_CURRENT_SOURCE_DIR}/impl/frontend/resources.cpp
  ${CMAKE_CURRENT_SOURCE_DIR}/impl/frontend/tokenizer.cpp
  ${CMAKE_CURRENT_SOURCE_DIR}/impl/vision/bindings.cpp)

target_include_directories(ninfer_engine PRIVATE
  ${CMAKE_CURRENT_SOURCE_DIR}/export)
```

Paths remain explicit. Do not export a CMake source variable to the parent, create a helper target,
or hide the source list behind a function.

### 6.3 Exact-target CMake fragment

Replace the 27B `add_library` declaration with `target_sources(ninfer_engine PRIVATE ...)` containing
the existing package, load, program, schedule, state, and diagnostic source list. Add its `export/`
and required private implementation include paths to `ninfer_engine`.

The current exact target has no `.cu` translation units. Removing its archive therefore does not
require moving `ninfer_cuda_archive` to `ninfer_engine`; CUDA implementation/device-link ownership
continues to reside in `ninfer_core` and `ninfer_ops`.

### 6.4 Direct internal consumers

Tests, the target activation-dump tool, and the target-private MTP benchmark currently link the
exact-target archive. After the cutover they link `ninfer_engine` instead. Consumers that include an
exact package or `impl/` header add the exact `export/` or `impl/` path explicitly; those paths do
not become public include directories of `ninfer_engine`.

This keeps the installed/public surface limited to `include/ninfer/engine.h` and
`include/ninfer/types.h` while allowing explicitly target-private development products to reach
their intended seams.

## 7. Family frontend contracts

### 7.1 `frontend.h`

Move the complete move/copy/lifetime facades for:

- `qwen3_6::PreparedPrompt`;
- `qwen3_6::OutputSession`;
- `qwen3_6::Frontend`;
- the family `PublishedOutput` alias.

Keep implementation ownership opaque through out-of-line destructors/PIMPL where it exists today.
The header may expose the family token-domain constant needed by both the tokenizer and exact target;
target-specific padded output rows remain exact storage facts.

### 7.2 `prepared_prompt.h`

Move the fully owning represented prompt data:

- `PromptModality`;
- `VisionGrid`;
- `TokenSpan`;
- `VisionItem`;
- `PromptIdentity`;
- `PrepareStats`;
- `PreparedPromptData` and its position-axis/media helpers.

Provide one narrow `PreparedPromptAccess` used by exact Programs:

```cpp
class PreparedPromptAccess {
public:
    static const PreparedPromptData& view(const PreparedPrompt&);
    static PreparedPromptData take(PreparedPrompt&&);
};
```

`view` supports request planning; `take` performs the existing ownership handoff into `Program`.
Neither method accepts or returns a target identity. Exact `Program` classes are not friends of the
family facade.

### 7.3 Frontend implementation move

Move the current tokenizer, template renderer, processor, prompt construction, MRoPE, stop-policy,
output decoder, and output-session implementation together. The move permits include-path,
namespace, facade-access, and construction-entry changes only; represented behavior is unchanged.

Use namespace:

```cpp
namespace ninfer::targets::qwen3_6 { /* family contracts */ }
```

Private implementation helpers may use a nested implementation namespace, but must not retain the
27B source key in a type, filename, error message, or include path.

The focused small-tokenizer/media fixture remains a test-only construction seam in
`impl/frontend/test_access.h`. It does not become part of an exact package facade.

### 7.4 Frontend resources

`frontend_resources.h` owns two passive values:

```cpp
struct FrontendResourcePlan {
    artifact::ObjectHandle tokenizer_json;
    artifact::ObjectHandle tokenizer_config_json;
    artifact::ObjectHandle chat_template_jinja;
    artifact::ObjectHandle generation_config_json;
    artifact::ObjectHandle preprocessor_config_json;
    artifact::ObjectHandle video_preprocessor_config_json;
};

struct FrontendResources {
    std::string tokenizer_json;
    std::string tokenizer_config_json;
    std::string chat_template_jinja;
    std::string generation_config_json;
    std::string preprocessor_config_json;
    std::string video_preprocessor_config_json;
};
```

It also declares narrow family functions to:

1. require and retain the six canonical raw resources in an existing `artifact::Binder`;
2. take those retained bytes into one owning `FrontendResources` value after materialization;
3. construct the family `Frontend` from that value.

The family binding helper never calls `Binder::finish()`. Only the exact target can close the
complete artifact inventory after Text, MTP, Vision, draft-head, and frontend objects are all bound.

## 8. Exact 27B package after frontend extraction

### 8.1 Package facade

`package.h` includes the family frontend contract and replaces its local frontend class definitions
with:

```cpp
using Frontend       = qwen3_6::Frontend;
using PreparedPrompt = qwen3_6::PreparedPrompt;
using OutputSession  = qwen3_6::OutputSession;
```

The following remain exact complete facade types in the same header:

- `LoadPlan`;
- `LoadedModel`;
- `SequencePlan`;
- `RequestPlan`;
- `Program`.

There is no family base package and no templated common package implementation.

### 8.2 Program bridge

`Program::plan_request` obtains a const family data view through `PreparedPromptAccess::view`.
`Program::begin` consumes the family data through `PreparedPromptAccess::take`. Its private
`Program::Impl`, request planning, prefix logic, state transitions, and schedules remain unchanged.

### 8.3 Loaded model ownership

The exact `LoadedModelData` continues to own `artifact::MaterializedArtifact backing` before every
typed device view. It holds one `qwen3_6::FrontendResources` value and exact/composite weight values.
Family helpers return non-owning tensor/weight views into that exact backing; the family directory
does not introduce a second loaded-model or device-allocation owner.

`Package::make_frontend` calls the family construction entry with the retained family resources.

## 9. Engine prepared-prompt cutover

Delete `targets::PreparedValue` from `targets/registry.h`. `ActiveTarget` remains the exact-target
variant and later grows only when a complete target is registered.

Change the public envelope implementation to the fixed family value:

```cpp
class PreparedPrompt::Impl {
public:
    Impl(PromptSummary summary, double prepare_seconds,
         qwen3_6::PreparedPrompt prepared);

    PromptSummary summary;
    double prepare_seconds = 0.0;
    qwen3_6::PreparedPrompt value;
};
```

Then:

- remove the templated constructor and `std::in_place_type` construction;
- keep the summary/timing copy because generation reports them after consuming the family value;
- remove the exact `Prepared` type deduction in `Engine::generate`;
- remove `std::get_if` and the target-mismatch exception;
- construct the `OutputSession` from `prompt.impl_->value` and move that value directly into
  `runtime::run_one`.

This change does not alter `ActiveTarget` visitation or exact `Program` dispatch.

## 10. Generic typed artifact helpers

Moving passive Vision binding into the family directory must not duplicate the physical format/view
construction currently private to the 27B binder. Add one narrow generic helper under
`src/artifact/typed_binding.h/.cpp` and compile it into `ninfer_artifact`.

It may own only mechanisms derived from registered artifact/core formats:

- `NumericFormat` to `StorageLayout` mapping;
- requiring a named device tensor and adding it to the materialization plan;
- requiring a raw resource and retaining it on host;
- constructing a direct `Tensor` view over materialized bytes;
- constructing a contiguous or row-split two-dimensional `Weight` view from explicit format,
  rows, and columns.

It must not own:

- a Qwen object name or shape;
- Text, MTP, Vision, layer, expert, tokenizer, or target knowledge;
- target row slicing or semantic submatrix roles;
- draft-token validation;
- a target load plan or call to `Binder::finish()`.

The exact binder immediately adopts the helper for its existing direct/quantized views. Its
`row_view`, MLP split, attention/GDN subviews, draft-id handling, and topology accounting remain
target-private.

## 11. Passive common Vision extraction

### 11.1 Family config

`qwen3_6::VisionBackboneConfig` owns only facts shared by the two family checkpoints:

- 27 transformer layers;
- hidden width 1152;
- intermediate width 4304;
- 16 heads and 72-wide heads;
- patch input width 1536;
- merge factor 2 and merge unit 4;
- merger input/hidden width 4608;
- 2304 position rows;
- Vision rotary dimension/theta and norm epsilon.

It does not contain Text output width. In the 27B execution context,
`out_hidden = TextConfig::hidden` remains an exact-target fact.

### 11.2 Family plans and weights

Move passive plans/weights for:

- patch embedding, patch bias, and position embedding;
- all twelve immutable roles in each of the 27 Vision transformer blocks;
- merger `fc1`, `fc1_bias`, norm weight, and norm bias.

Keep these exact:

- merger `fc2` with shape `[5120,4608]`;
- merger `fc2_bias` with shape `[5120]`;
- the composite `VisionWeights` used by the 27B schedule.

The exact composite has the conceptual shape:

```cpp
struct VisionWeights {
    qwen3_6::VisionCommonWeights common;
    Weight merger_fc2;
    Tensor merger_fc2_bias;
};
```

### 11.3 Binding order

Preserve the current materialization order without making target width a family parameter:

1. family binds the stem and 27 backbone layers;
2. family binds merger `fc1` and `fc1_bias`;
3. exact 27B binds merger `fc2` and `fc2_bias`;
4. family binds merger norm weight and bias;
5. exact 27B binds no remaining Vision object and eventually closes the whole artifact with
   `Binder::finish()`.

Use narrow `bind_vision_backbone`, `bind_vision_merger_input`, and `bind_vision_merger_norm`
functions rather than a callback, template, runtime dimension parameter, or generic Vision graph.

### 11.4 Exact schedule rewiring

Keep every source under `impl/schedule/vision*`, `impl/program/`, and `impl/state/` in place. Update
only include paths, config references, and passive weight member paths, for example:

```text
weights.vision.layers[i]
  -> weights.vision.common.layers[i]
```

Do not change Op calls, launch order, graph capture, workspace sizing, patch/control transforms,
visual scatter, merger execution, or Text injection.

## 12. Implementation phases

Each phase must leave the tree buildable. Do not combine a source-composition failure with a
frontend or Vision behavior change.

### Phase A: source aggregation with no ownership move

1. Move `ninfer_engine` creation before the target subdirectory calls.
2. Convert the existing 27B CMake file from `add_library` to `target_sources(ninfer_engine ...)`.
3. Centralize the aggregate engine dependencies listed in Section 6.
4. Remove the exact-target CMake target and its CUDA-archive call.
5. Repoint direct target tests, diagnostic, and benchmark to `ninfer_engine` and add their internal
   include paths explicitly.
6. Configure and build the engine, diagnostic, benchmark, and affected tests before moving a source
   file.

Acceptance for Phase A: the same 27B implementation runs from one `ninfer_engine` archive, and no
`ninfer_qwen3_6_27b_rtx5090` CMake target remains.

### Phase B: family frontend and resource extraction

1. Create the family source/export tree and its `target_sources` CMake fragment.
2. Move the three frontend facades from exact `package.h` to family `frontend.h`.
3. Move prepared data and access to `prepared_prompt.h`.
4. Move the six resource handles/strings and their binding/materialization functions to
   `frontend_resources.h` and `impl/frontend/resources.cpp`.
5. Move the tokenizer/template/processor/frontend implementation files and change namespaces and
   includes.
6. Replace exact facade definitions with aliases and switch `Package::make_frontend` to the family
   construction entry.
7. Rewire Program/schedule includes to the family prepared-data contract.
8. Build and run the focused frontend tests plus a full `ninfer_engine` link.

Acceptance for Phase B: the exact target has no `impl/frontend/` directory, and its frontend behavior
comes only from `src/targets/qwen3_6/`.

### Phase C: direct Engine prompt ownership

1. Delete `PreparedValue` from the registry.
2. Store `qwen3_6::PreparedPrompt` directly in the public PIMPL.
3. Remove exact-type construction, extraction, and mismatch logic.
4. Build public API, generation-controller, frontend, and real Engine integration tests.

Acceptance for Phase C: only `ActiveTarget` remains variant-based; prepared prompts contain no
exact-target alternative or identity.

### Phase D: generic typed binding helper

1. Add the narrow artifact helper and register it in `ninfer_artifact`.
2. Replace the equivalent private mechanisms in the 27B binder.
3. Keep semantic target views and validation in the exact binder.
4. Run artifact reader/materialization tests and load the real 27B artifact.

Acceptance for Phase D: family Vision can form typed views without importing exact-target code, and
no checkpoint fact has entered `src/artifact`.

### Phase E: passive Vision extraction

1. Move common Vision constants to family `vision.h`.
2. Move common Vision plan/weight types and binding/materialization definitions.
3. Leave the target-width projection in exact load code.
4. Rewire exact schedule member paths without changing execution.
5. Build all Vision callers and run the real multimodal Engine path.

Acceptance for Phase E: family owns passive common Vision definitions while every Vision execution
method remains exact-target code.

### Phase F: tests and active authorities

1. Move the focused C++ frontend test to `tests/targets/qwen3_6/test_frontend.cpp` and rename its
   CMake test target to `ninfer_qwen3_6_frontend_test`.
2. Link that test to `ninfer_engine`; expose only the family private test-support include path it
   actually requires.
3. Remove the single `vision_workspace_bytes(...) > 0` assertion from the family frontend test. It
   is target scheduling, and the real multimodal Engine integration is the meaningful retained
   check.
4. Keep state-store, visual-scatter, prefix/MTP, activation-dump, and target benchmark code under
   the 27B identity while linking their binaries to `ninfer_engine`.
5. Update `docs/design.md` and `docs/ninfer-engine-architecture.md` from separate family/exact static
   targets to the single-engine source-composition model.
6. Update implementation maps and test documentation only where file ownership or build commands
   changed. Describe the final state only; do not record migration provenance.
7. After implementation and verification are complete, move this plan to `docs/archive/` and update
   both document indexes in the same change.
