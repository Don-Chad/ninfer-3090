# RTX 3090 Qwen3.6-35B-A3B text-only inference

The complete 20.84 GiB Qwen3.6-35B-A3B NInfer artifact now runs on a 24 GiB RTX 3090 under native
Windows. No weights were removed or requantized. The artifact retains its mixed Q4 routed
gate/up, mostly-Q5 routed down, Q6 full head, optimized Q4 draft head, and W8 always-active/MTP
weights.

The required `--text-only` engine option rejects image/video prompts and omits the maximum vision
scratch reservation. At a 128-token capacity this reduced workspace capacity from roughly 1.90
GiB to 15.05 MiB. The verified smoke run reserved 20.96 GiB total: 20.82 GiB weights, 126.04 MiB
sequence state, and 15.05 MiB workspace.

## Recommended Windows command

```powershell
build-windows\apps\Release\ninfer.exe models\qwen3_6_35b_a3b.ninfer `
  --prompt "Explain speculative decoding in three sentences." `
  --max-context 256 --prefill-chunk 128 --max-new 128 `
  --kv-dtype int8 --mtp-draft-tokens 2 --lm-head-draft --text-only
```

CUDA Graphs are enabled by default and should remain enabled. The current matched sweep selects
MTP K=2 for the measured corpus; K=3 is close enough that real workloads with better third-position
acceptance can still favor it.

The repeated results use tg128, CUDA Graphs, INT8 group-64 KV, a 128-token prefill chunk, a
256-token engine capacity, and five measured repetitions after two warm-ups.

These results use the rebuilt checkpointed source. They supersede the initial 32-token sweep,
which selected K=1 before the later small-T and projection work. K=3 obtains more output per

gain on this corpus.

## Real CLI generation

The controlled tg128 results above are the best numbers for comparing kernel revisions. A second
benchmark uses the actual streaming CLI, sampling, model-generated thinking, stop-token handling,
and the exact prompt `Explain prefill and decode in three sentences.`. These are single real
generations rather than repeated fixed-token microbenchmarks.

| Runtime | Context capacity | MTP | Generated | TTFT | Decode tok/s | Overall tok/s | Acceptance |
|---|---:|---:|---:|---:|---:|---:|---:|
| Native Windows v2 | 4,096 | disabled | 619 | **41 ms** | **186.81** | 184.86 | n/a |
| Native Windows v2 | 4,096 | K=3 + LM-head draft | 1,065 | **40 ms** | **284.75** | 281.99 | 70.57% |

Both runs use the same 4,096-token capacity, sampler, 32,768-token generation ceiling, INT8 KV,
CUDA Graphs, `--text-only`, prompt, and native Windows runtime. TTFT is measured from the start of
request preparation until the first non-empty streamed reasoning or content delta; one-time model
loading and CUDA Graph construction are excluded. The different generated sequences mean the
decode rows remain real operational samples rather than a controlled fixed-token MTP speedup.
Observed one-time engine construction was 4.713 s for normal mode and 4.885 s for MTP-3, making
cold-process launch to first streamed token approximately 4.754 s and 4.925 s respectively.

Full v2 generated text and diagnostics are preserved in
`benchmark-results/35b-cli-ctx4096-k0-v2-windows.txt` and
`benchmark-results/35b-cli-ctx4096-k3-v2-windows.txt`.

## RTX 3090 kernel work

An Nsight Systems node trace of MTP K=1 identified the Q4 routed gate/up, Q5 routed down, W8 GDN
input, W8 row-split projection, and Q6 verification head as the largest kernel groups. The Q6
K=2048 head was isolated with the CUDA-event linear harness. Its T=2 cold-cache latency improved
from 740.35 us to 669.70 us by using triple-buffered cp.async staging, L2-only (`cg`) code loads,
and a two-block launch bound on `sm_86`.

That schedule regressed the dense model's K=5120 multi-column head, so dispatch is deliberately
shape-specific: K=2048 uses the GA102-tuned schedule and K=5120 retains its original schedule.
The operator improvement produced an end-to-end MTP gain from 249.99 to 252.83 tok/s (1.14%) and
an initial ordinary-decode gain from 178.11 to 179.09 tok/s (0.55%). A second normal-only pass
selected balanced eight-warp Q4 gate/up and Q5 down schedules while retaining the stable nine-warp
Q6 down route. It raised the ten-run tg128 mean to 186.26 tok/s (187.09 median), approximately 4%
above the previous 179.09 tok/s controlled result. MTP routing was deliberately left unchanged.
The 4K TTFT run also exposed an undersized ordinary-only CUDA Graph reservation. The planner now
reserves 28 MiB for each ordinary graph executable, covering the measured 101,707,776-byte cold
capture while keeping CUDA Graphs enabled.

Runtime v2 doubles the contiguous W8 K=2048 work per lane from eight to sixteen values. This cuts
the number of decode phases and scale shuffles in half while retaining coalesced code and
activation traffic. The attention input projection improved from 33.792 to 32.768 us (3.0%); the
larger GDN projection remained at 39.936 us. In the matched 4K tg128 benchmark, the ten-run normal
mean improved from 186.94 to 188.61 tok/s (0.89%), with a 188.59 median and 0.30 tok/s standard
deviation. Shared-activation staging, a 32-value no-shuffle form, and a reduced router shuffle were
measured and rejected because they regressed latency.

Raw JSON, CSV, smoke logs, and Nsight reports are under `benchmark-results/` in the working tree.
