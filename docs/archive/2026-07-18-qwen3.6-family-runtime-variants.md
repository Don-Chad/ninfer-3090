# Qwen3.6 Family Runtime Variants Refactor Plan

> Status: completed and ready for archive.
>
> Current implementation status: both `qwen3_6_27b_rtx5090` and
> `qwen3_6_35b_a3b_rtx5090` are compiled closed-registry alternatives over one family runtime.
>
> This plan replaces the independent 35B execution-package design. It first moves the duplicated
> Qwen3.6 execution mechanisms into one family runtime, migrates 27B onto that runtime, and then
> adds 35B-A3B as a peer variant.

Completion evidence: the full build, family mechanism check, real 27B Engine route, and real 35B
Engine route (including Text/MTP/prefix/Vision and native 262144-context construction) pass. One
public 35B `pp1024+tg64`, graph-enabled K=3 baseline recorded 550.35 prefill tok/s and 251.48 output
tok/s on the RTX 5090. Three read-only old/new audits found no remaining Program/state, Text/MTP,
dimension/binding, Vision, ownership, or registration blocker.

## 1. Required outcome

The completed source tree has one Qwen3.6 runtime implementation and two symmetric compiled
variants:

```text
Qwen3.6 family runtime
├── qwen3_6_27b_rtx5090 variant
└── qwen3_6_35b_a3b_rtx5090 variant
```

Neither variant imports, extends, or falls back to the other. The closed registry still selects an
exact checkpoint/GPU package by `model_id`, and each selected package owns its immutable weights and
one mutable runtime instance. The sharing is source-level, compile-time Qwen3.6 reuse; it is not a
runtime model graph or a new public abstraction.

Completion requires:

- common Program, request, Text/MTP/Vision orchestration, state-transition, planning, and CUDA Graph
  mechanisms live once under `src/targets/qwen3_6`;
- 27B and 35B-A3B each provide only their exact identity, configuration, bindings, execution leaves,
  and graph frontier-range data;
- the migrated 27B target preserves its supported public behavior;
- the 35B-A3B artifact becomes the second fully registered exact target through the same Engine
  route; and
- obsolete duplicated target schedules are removed rather than retained as compatibility paths.

This work does not redesign an Op, artifact format, serving protocol, or public C++ API.

“Fully registered” is part of this plan, not follow-up work. At completion,
`model_id="qwen3.6-35b-a3b"` selects target key `qwen3_6_35b_a3b_rtx5090` from the closed registry;
the package is compiled into `ninfer_engine`; all Package factories are wired; Engine construction,
frontend preparation and generation succeed from the named artifact; and CLI, serving and benchmark
reach it through their existing Engine construction path without target-specific adapters. Delivered
capability documentation lists the target.

## 2. Normative architecture contract

This section fixes ownership for the refactor. “Family” and “Variant” below are normative semantic
boundaries, not suggestions about code placement. Mechanical choices such as splitting a `.cpp`
file do not change them.

### 2.1 Composition and runtime types

The registry selects an exact package once. The package then aliases the following Qwen3.6 family
types with its private compile-time variant:

```text
artifact.model_id
  -> closed exact-package registry
       -> qwen3_6_27b_rtx5090::Package
            SequencePlan = qwen3_6::SequencePlan<Variant27B>
            RequestPlan  = qwen3_6::RequestPlan<Variant27B>
            Program      = qwen3_6::Program<Variant27B>
       -> qwen3_6_35b_a3b_rtx5090::Package
            SequencePlan = qwen3_6::SequencePlan<Variant35BA3B>
            RequestPlan  = qwen3_6::RequestPlan<Variant35BA3B>
            Program      = qwen3_6::Program<Variant35BA3B>
```

These three template types, rather than a separate class named `Runtime`, are the family runtime.
Their API is the existing package/runtime contract. The exact package keeps `LoadPlan`,
`LoadedModel`, `model_id`, preflight, load/materialization, frontend construction, and the coarse
factory functions. It does not keep wrapper implementations of the three family types.

`package.h` forward-declares its Variant and aliases the family types. The Variant is completed in
the exact target's private implementation and explicitly instantiated there. Family sources never
include an exact-target header. There is no virtual base, runtime target enum, `model_id` branch,
phase registry, or sibling-target dependency.

Registration covers the whole exact package, not just a Variant instantiation: the build owns its
sources, the registry dispatches its artifact identity, and `plan_load`, `construct_loaded_model`,
`make_frontend`, `plan_sequence`, and `create_program` all resolve before public Engine execution.
There is no “loadable but unregistered” or “runtime-only” 35B state accepted by this plan.

The selected exact package owns the `Program<Variant>` object and therefore its persistent arena,
workspace arena, graphs, host staging, and mutable request state. The family owns the algorithms
and state types; no bytes or state are shared between two Program instances.

### 2.2 Exact package, family, and controller boundary

| Type or responsibility | Single owner | Fixed meaning |
|---|---|---|
| `LoadPlan`, `LoadedModel` | exact package | exact artifact inventory, aliases, materialization and immutable typed weights |
| `SequencePlan<Variant>` | family | validated capacity/options plus persistent, workspace and graph layouts derived from Variant facts |
| `RequestPlan<Variant>` | family | read-only request admission, reuse path, Vision spans, MTP preparation and transient requirement; owns no pointer into a prepared prompt |
| `Program<Variant>` | family algorithm; exact instance owns bytes | request lifecycle, Text/Vision/MTP execution, state transaction, prefix reuse, diagnostics and summaries |
| `PreparedPrompt`, `OutputSession` | family frontend | identity-free owning Qwen3.6 input and output semantics |
| generation controller | `src/runtime` | output-token budget, stop/cancellation decisions, `OutputSession` preview/commit and user-visible publication |

`Program<Variant>` retains exactly the current controller-facing operations:

```text
plan_request, begin, decode_round, resolve_pending,
finish_active, abort_request, materialized_tokens,
memory_summary, speculative_stats, generation_timings, reset_memory_peaks
```

`begin` and `decode_round` produce `runtime::BeginResult`/`GeneratedRound`; `resolve_pending`
commits only the accepted model-state prefix. The family Program must not interpret stop strings,
publish output, or own an output budget. Conversely, `src/runtime` must not inspect KV/GDN slots,
MTP proposal state, graph ranges, or target weights.

Exact `LoadedModel` owns the materialized artifact backing and an immutable `Variant::ModelView`.
The exact Package factories pass that view to family sequence planning and Program construction;
the family stores the view but never the Binder, artifact inventory, or exact `LoadedModel` type.
The existing Engine lifetime keeps `LoadedModel` alive longer than its Program.

### 2.3 Fixed family-owned mechanisms

The following implementation exists once under `src/targets/qwen3_6` and is not overridable by a
Variant:

| Area | Fixed family implementation |
|---|---|
| Frontend | tokenizer, chat template, interpretation of the six artifact-backed resources, media preprocessing, prepared-prompt construction, MRoPE prompt positions and output-session behavior; each exact package supplies its own six resource byte spans |
| Semantic weight views | identity-free Qwen3.6 layer/MTP/Vision view schemas; exact binders populate them and attach only the leaf payloads defined in Section 2.4 |
| Hybrid decoder | the three-GDN/one-full-attention topology and compact-index formulas, layer traversal, offset pre-norm and residual ordering; layer count and widths are Variant facts |
| Text root/tail | token or composed-embedding input, chunk traversal, scalar/three-axis positions, final norm, full/draft head invocation, token-domain restriction, sampling and returned-round assembly |
| Full attention after projection | Q/K offset RMSNorm, RoPE, GQA append/attention, sigmoid gate, output projection and residual update |
| GDN after projection | ordinary/snapshot causal convolution, Q/K/V extraction, L2 normalization, ordinary/snapshot gated-delta recurrence, gated RMSNorm, output projection and residual update |
| Post-mixer placement | one post-mixer call after every main decoder mixer and every complete MTP layer; only the call's closed leaf is Variant-owned |
| Text execution modes | prompt prefill, ordinary one-token decode and `K+1` target verification, including all workspace scopes and state-view selection |
| MTP | shifted composed-input alignment, cross-chunk bridge, KV-only aligned prefill, initial proposal, autoregressive draft, target verify, acceptance, rebuild, accepted-row gather and ordinary fallback |
| Vision | one item-bounded backbone executor, ordered item consumption, merger scatter into Text, and the shifted MTP consumer |
| Program state | `Empty/Active/Pending/Resident/Invalid` lifecycle, `0 <= E <= capacity` and `E <= S <= E+1`, ledger, `rope_delta`, KV/GDN frontiers and slots, pending-candidate transaction, boundary snapshot, reset/abort and append/prefix reuse |
| Planning | persistent layout composition, request admission, request transient layout, common workspace regimes and graph allowance calculation |
| CUDA Graphs | warm/capture/replay mechanics, graph-stable controls and addresses, ordinary/aligned/MTP graph construction, range selection and gap-free coverage validation |
| Diagnostics | semantic activation tap positions, common lifecycle/state reporting and common timing/stat aggregation |

Changing tensor extents, layer counts, weight formats, or the concrete Op route selected by a
public Op does not move any row in this table into the Variant.

### 2.4 Closed Variant contract

Outside the Variant interface, each exact package owns its `model_id`, RTX 5090 preflight,
artifact inventory/binding, load summary and target-labelled diagnostic/reference entry points.
Those values are never passed into family scheduling code.

The Variant visible to the family supplies only the following data:

- dimensions, counts, limits, scales, codec/layout facts and supported option values;
- immutable embedding, norm, output-projection, head, convolution, recurrence, MTP and Vision
  weight views used directly by the family algorithm;
- target-private binding payload types for the three leaves below;
- persistent/admission constants, including native-context and memory ceilings;
- fixed queries returning ordinary and MTP frontier ranges; each Variant always supplies these data,
  although it may call a family helper to construct common ranges.

There are exactly three Variant execution-leaf families:

| Leaf | Input and output contract | 27B implementation | 35B-A3B implementation |
|---|---|---|---|
| `AttentionProjection` | normalized BF16 `h[H,T]` to BF16 `q[D,Nq,T]`, `gate[D,Nq,T]`, `k[D,Nkv,T]`, `v[D,Nkv,T]`; its MTP form also exposes KV-only and final-column Q/gate-only entry points for aligned prefill | consume split QK and gate/value parents, or the exact MTP packed binding | consume the fused main-attention parent, or the exact MTP packed binding |
| `GdnProjectionControl` | normalized BF16 `h[H,T]` to pre-convolution BF16 `qkv[C,T]`, BF16 output gate `z[V,T]`, and FP32 `g[Hv,T]`/`beta[Hv,T]` | existing QKV projection, gating projection and Z projection | fused QKV/Z projection and fused A/B control projection |
| `PostMixer` | normalized BF16 `h[H,T]` plus BF16 residual `x[H,T]`; update `x` in place with the complete post-mixer result | dense SwiGLU and down projection for the applicable main/MTP binding | routed/shared sparse MoE with `AddResidual` for the applicable main/MTP binding |

Main Text and MTP may use different private binding payload types because their physical weights
differ. They are overloads of the same semantic `AttentionProjection` or `PostMixer` leaf, not new
schedules. The MTP attention leaf's complete, KV-only, and Q/gate-only entry points are fixed by the
family MTP algorithm; a Variant cannot add another projection mode. `GdnProjectionControl` applies
only to GDN main layers because the admitted MTP layer is full attention.

The GDN leaf family has exactly three staged entry points, always called by the family in this
order: pre-convolution QKV plus the already-allocated optional Z output; post-convolution FP32
`g/beta`; and post-recurrence Z production or no-op. The 27B Variant fills QKV first, control second,
and Z third. The 35B-A3B Variant fills fused QKV/Z first, fused A/B control second, and makes the
third entry point a no-op. These are fixed entry points of one leaf family, not three additional
leaves or a target-selectable schedule.

Each leaf entry point has a matching workspace-layout contribution taking the same token extent and
binding kind. It may reserve only temporaries used inside that leaf. The family owns the
input/output tensor allocations shown in the table, the surrounding scope, and all non-leaf Op
work. Planning and execution must call the corresponding pair; hard-coded duplicate byte formulas
are forbidden.

The family-owned semantic layer views contain common weights plus an exact leaf payload. Family
code reads the common weights and can only pass the payload to its matching Variant leaf. A Variant
leaf receives no Program, decoder state, lifecycle, positions, KV view, MTP transaction, Vision
control, or graph object, so it cannot grow into a target-private schedule.

The list is closed. A fourth leaf, a new phase callback, or a target-dependent branch in family
code requires an explicit architecture-document amendment based on a demonstrated mathematical or
storage difference before implementation. “Convenient for this target” is not sufficient.

### 2.5 Vision, workspace, and graph rules

The shared Vision lifetime is fixed:

1. `RequestPlan` builds one owning `VisionControl` and ordered Text/media spans.
2. The family encodes each item immediately before its first consuming span.
3. One item's merger output remains live until both its final Text scatter and its final shifted-MTP
   consumer have been enqueued on the Program stream.
4. Only then may its transient region be reused for the next item; no host round trip or per-item
   synchronization is introduced.
5. A chunk never contains columns from two media items. One media item may span multiple chunks,
   and adjacent ordinary text may share a chunk with that one item.

The family plans these mutually scoped workspace regimes and takes their maximum rather than their
sum: Text prefill; ordinary decode; target verify/MTP rebuild; MTP KV-only prefill/final tail;
sampling/acceptance; and one Vision item. Variant workspace contributions appear only at the three
leaf call sites. Persistent KV/GDN/round state and request-lifetime item output are planned outside
the reusable workspace arena.

Each Variant's graph-range queries supply data only: ordered, inclusive frontier ranges for ordinary
and MTP graphs. The family owns capture contents, stable addresses, warmup, replay and selection.
When the corresponding graph mode is enabled and admitted, it rejects empty, overlapping,
reversed, or gap-producing coverage of that mode's derived frontier domain. A different threshold
is a Variant value; a different capture schedule is not a Variant hook.

### 2.6 Prohibited ownership drift

- The family runtime is specific to the two admitted Qwen3.6 variants; it is not a generic model
  framework or runtime graph.
- Exact artifact binding stays exact-target-owned, and neither target imports, wraps, or falls back
  to its sibling.
- Family execution calls only public repository-internal Op contracts. Mathematical kernels and
  private Op plans remain in `src/ops`.
- Registry selection surrounds the complete package. There is no per-layer target dispatch.
- Prepared prompts carry no exact-target identity or mismatch machinery.
- Exact target code may not define Text, Vision, MTP, Program-lifecycle, state-transition,
  workspace-composition, or graph-capture methods.
- Transitional wrappers and copied schedules are deleted with the 27B migration; they are not a
  compatibility lane.

## 3. Source and instantiation shape

`src/targets/qwen3_6` gains the repository-internal declarations for `SequencePlan<Variant>`,
`RequestPlan<Variant>`, `Program<Variant>`, the semantic weight views, and the private common
planning/Text/MTP/Vision/graph implementations. Existing frontend, state and passive Vision types
remain there.

Each exact directory retains only:

```text
export/.../package.h        exact identity/load facade and aliases to family runtime types
impl/package.cpp            preflight, load/front-end factories and Program construction
impl/config.*               exact dimensions, limits, scales and option facts
impl/variant.*              three leaves, their workspace contributions and graph-range data
impl/load/bindings.*        exact artifact validation, materialization and semantic views
impl/diagnostic/*           exact labels and reference adapter; no execution schedule
```

Template definitions stay in family-private headers and are explicitly instantiated by each exact
target; the family never includes an exact header. `ninfer_engine` remains the sole compile owner.
File splitting is not an acceptance criterion, but the ownership and closed interface in Section 2
are.

## 4. Ordered implementation

These are three coherent changes, not validation gates or partial product lanes.

### 4.1 Land the authority change

In one documentation change:

1. Replace the current no-family-execution boundary in `AGENTS.md`, `docs/design.md`, and
   `docs/ninfer-engine-architecture.md` with Section 2 of this plan.
2. Update `docs/qwen3.6-27b-architecture.md` and `docs/qwen3.6-35b-a3b-architecture.md` so their
   implementation maps assign the fixed algorithm to the family and only the closed leaves to the
   exact target.
3. Update artifact documents only where they describe runtime ownership; their storage inventory,
   formats, aliases and conversion authority remain exact-target-specific.
4. Keep all capability documents explicit that only 27B is registered and mark the family
   migration as pending. Advertise 35B only after Section 4.3 lands.
5. Keep this file as the only active implementation plan and remove links to the superseded
   independent-35B plan.

### 4.2 Migrate 27B first

Use the working 27B target to establish the family runtime:

1. Define the three family runtime types and semantic weight views, then move the existing 27B
   planning, Program, Text, MTP, Vision and graph algorithms into them without changing operation
   order or state semantics.
2. Make the 27B binder populate the family views. Implement `Variant27B` with its current exact
   configuration, main/MTP projection payloads, the three closed leaves and their workspace
   contributions, and its current ordinary/MTP graph frontier ranges.
3. Replace the exact `SequencePlan`, `RequestPlan`, and `Program` PIMPL classes with the required
   Package aliases to the family instantiation. Keep exact `LoadPlan`, `LoadedModel`, `model_id`,
   preflight and factories.
4. Move Vision execution to the fixed item lifetime in Section 2.5. The 27B binder supplies the
   5120-wide merger-output view; it supplies no Vision schedule hook.
5. Explicitly instantiate the family types for `Variant27B`, update CMake ownership, and delete the
   superseded exact `program/` and `schedule/` sources in the same change. Retain only the Package
   aliases required by Section 2.1; do not retain exact wrapper classes or a legacy path.

### 4.3 Add the 35B-A3B peer variant

Add `Variant35BA3B` and its exact package:

1. Add the exact package and bind the named
   `out/qwen3_6_35b_a3b_rtx5090.ninfer` artifact's accepted 883 tensors and six resources into the
   family semantic views and private leaf payloads.
2. Implement `Variant35BA3B` with the 2048-wide configuration, fused attention and GDN leaves,
   sparse-MoE post-mixer leaf, their workspace contributions, native context/memory limits, and
   complete gap-free graph ranges matching the qualified 35B GQA routes.
3. Explicitly instantiate the same three family runtime types. No Program, Text, MTP, Vision,
   workspace-composition, or graph-capture source is added under the 35B directory.
4. Add the exact package sources and explicit instantiation to the `ninfer_engine` build, then add
   the closed-registry alternative mapping `qwen3.6-35b-a3b` to
   `qwen3_6_35b_a3b_rtx5090::Package`.
5. Wire every Package factory and verify the target's load summary, frontend and Program are reached
   through public Engine construction. CLI, serving and benchmark receive no target branch; their
   existing Engine artifact option makes the registered target available.
6. Update delivered-capability documentation and archive this plan after the focused evidence in
   Section 5 passes.

An implementation discovery outside the closed contract is not resolved ad hoc. Stop that part of
the implementation, record the concrete mathematical or storage difference, and amend the active
architecture boundary before adding a leaf or moving an algorithm.

Run only the following evidence:

| Point | Required evidence |

| 35B registration and public behavior | add only `tests/targets/qwen3_6_35b_a3b/test_engine_real.cpp` as `ninfer_qwen3_6_35b_a3b_real_test`; construct public Engine from the named artifact, assert target `qwen3_6_35b_a3b_rtx5090`, 883 tensors, six resources and the exact materialized bytes, then use `max_context=4096`, `prefill_chunk=1024`, graph enabled and `K=3` to cover the fixed greedy prompt, an ordinary fallback, MTP resolution, one append/prefix-reuse request, and the inline PPM request. | 35B maximum configuration | construct once with `max_context=262144`, INT8-G64 KV, `K=5`, `prefill_chunk=1024` and graphs enabled; allocation must succeed, its memory summary must agree with the planned layout, and an over-capacity request must be rejected before Program mutation |

Existing independent Op oracles remain the numerical authority for the three leaves and individual
Ops. Cross-runtime or cross-schedule generated-token identity is not a numerical contract and is
not an acceptance test. Do not rerun every Op shape, layer tap, MTP window, graph range, media
combination, server adapter, or benchmark corpus. Use activation taps, NSYS, or NCU only to localize
a failure in the listed evidence. After correctness passes, record one public-Engine 35B baseline
with a 1024-token prefill, 64 requested outputs, graph enabled and `K=3`. It has no pass threshold
and does not trigger a profiling campaign unless execution selected a route forbidden by Section
2.4.

## 6. Completion criteria

The work is complete when:

1. active architecture documents describe one Qwen3.6 family runtime with two peer variants;
2. common execution/state/planning source exists once under `src/targets/qwen3_6`;
3. neither exact target imports the other or retains a copied family schedule;
4. the migrated 27B target passes its retained public Engine integration behavior;
5. `qwen3.6-35b-a3b` is compiled into and selected by the closed Engine registry as
   `qwen3_6_35b_a3b_rtx5090`, every Package factory is live, and the target completes Text, MTP,
   multimodal, prefix/state, graph, and memory behavior through the same family runtime;
6. CLI, serving and benchmark can select the 35B artifact through their unchanged public Engine
   route, and delivered-capability documentation lists it; and
7. the focused evidence in Section 5 passes, with any unavailable check stated explicitly.
